Define when comparison is required:
* two or more candidates address the same capability
* a candidate overlaps a local approved or incubating artifact
* multiple versions or forks exist
* a candidate may supersede an existing artifact
* a skill and a workflow solve similar problems differently

Required comparison dimensions:
* responsibility
* artifact type
* activation conditions
* expected inputs
* expected outputs
* side effects
* execution boundary
* authoritative source
* immutable version
* license quality
* attribution obligations
* maintenance activity
* security risk
* prompt-injection exposure
* platform compatibility
* portability
* prompt or context size
* scripts and dependencies
* overlap with local artifacts
* evaluation evidence
* adaptation burden
* long-term maintenance burden

Define comparison outcomes:
* prefer_candidate_a
* prefer_candidate_b
* keep_both_complementary
* merge_concepts_only
* adapt_one
* reject_both
* defer
* insufficient_evidence

Clarify:
* Stars, downloads, marketplace rankings, endorsements, or social popularity are not decisive quality or safety evidence.
* A shorter skill is not automatically better.
* A larger skill is not automatically more complete.
* License or safety blockers override feature advantages.
* Comparison does not authorize import.
